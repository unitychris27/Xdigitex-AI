# AI Development & Automation Platform – Revised Product Instructions

## Core Principle

The platform is AI-first.

Users should not be required to:

* Choose programming languages
* Choose frameworks
* Create folder structures
* Configure stacks manually
* Select databases manually

The primary workflow is:

User Prompt → AI Builds → Files Generated → Preview → Deploy/Publish

The AI is responsible for determining architecture, technologies, dependencies, and project structure.

---

# Project Creation

Projects are created automatically from AI workspace builds.

A project is simply a container for everything the AI has generated.

Example:

User:

"Build a SaaS invoicing platform with subscriptions and team management."

System:

* Creates Project
* Starts Build
* Generates Files
* Stores Build Outputs

The generated application automatically becomes a project.

Users should never have to manually create empty projects.

---

# AI Workspace

The AI Workspace is the primary interface.

Users interact through prompts.

Example prompts:

* Build a CRM SaaS
* Create a Telegram automation
* Build a Laravel API
* Build a Next.js ecommerce site
* Create a Django backend
* Create a VPS monitoring automation
* Build an AI support platform

The AI analyzes requirements and generates complete solutions.

---

# Stack Selection

Remove all manual technology selectors.

Do NOT show:

* Language Dropdown
* Framework Dropdown
* Database Dropdown
* Frontend Framework Selector
* Backend Framework Selector

The AI determines the technology stack automatically.

Example:

Prompt:

"Build an ecommerce website."

AI may choose:

* Next.js
* PostgreSQL
* Prisma
* Tailwind
* Docker

Prompt:

"Build a Laravel CRM."

AI must use Laravel.

Prompt:

"Build a FastAPI backend."

AI must use FastAPI.

If the user specifies technologies, the AI must follow them.

If the user does not specify technologies, the AI is free to choose the most appropriate stack.

---

# AI Build Modes

Users should not select AI providers directly.

Do NOT expose:

* OpenAI
* Anthropic
* DeepSeek
* Gemini
* Grok

Instead expose build quality modes.

## Economy

Lowest cost mode.

Uses lower-cost providers.

Examples:

* DeepSeek
* Gemini Flash
* Other budget models

Best for:

* Landing pages
* Small projects
* Testing ideas
* Draft generation

---

## Balanced

Default mode.

Uses mixed model routing.

Optimized for:

* Quality
* Speed
* Cost

---

## High Power

Premium build mode.

Uses strongest reasoning models.

Primary provider:

* Anthropic Claude

Can also include:

* Premium GPT models
* Premium reasoning models

Best for:

* Large SaaS platforms
* Complex architectures
* Enterprise systems
* Advanced automations

High Power consumes more tokens and costs more credits.

---

# AI Build Process

When a user submits a prompt:

1. Analyze Requirements
2. Determine Architecture
3. Select Technologies
4. Generate Project Structure
5. Generate Code
6. Generate Documentation
7. Generate Configuration Files
8. Generate Deployment Files
9. Create Preview
10. Create Download Package

The entire process should be visible in real time.

---

# Files Management

Every file generated by AI must automatically appear in the Files section.

Examples:

frontend/
backend/
database/
docker/

README.md
.env.example
docker-compose.yml

No manual file creation required.

The Files section always reflects the latest generated build.

---

# ZIP Package Generation

Every successful build must automatically generate a downloadable ZIP package.

Example:

project.zip

Actions:

* Download ZIP
* Download Source Code
* Export Project

Users should always be able to obtain the complete generated source code.

---

# Preview System

Preview should be a dedicated tab.

Do NOT display Preview beside the workspace.

Incorrect Layout:

Editor | Preview

Correct Layout:

Chat | Files | Preview | Deploy | History | Logs

The Preview tab should render the latest generated output.

Examples:

Web Applications:
Render website.

Dashboards:
Render dashboard.

Landing Pages:
Render page.

APIs:
Show API documentation.

Automations:
Show workflow visualization.

Mobile Apps:
Show emulator preview when available.

The Preview tab must always reflect the latest generated build.

---

# Deploy & Publish

After a successful build, users should immediately see:

Deploy

or

Publish

buttons.

The goal is:

Build → Preview → Deploy

instead of:

Build → Download → Configure → Deploy

# Project Structure

Everything generated by AI belongs to the project.

Each project contains:

* Chat History
* Generated Files
* ZIP Packages
* Build History
* Deployments
* Logs
* Automations
* Assets

Projects become the permanent home of everything created within the AI Workspace.

Chat

Files

Preview

Deploy

History

Logs

Everything begins with a prompt and ends with a deployable solution.

# Product Positioning

The platform is not a website builder.

The platform is not a bot builder.

The platform is an AI Development & Automation Platform.

Users can:

* Build software
* Build SaaS products
* Build APIs
* Build mobile apps
* Create automations
* Connect Telegram
* Connect servers
* Execute SSH operations
* Deploy applications
* Manage workflows
* Create AI agents

The AI handles the technical implementation while the user focuses on describing what they want built.
